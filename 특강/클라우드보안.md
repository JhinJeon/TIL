# 자동화 테스팅 기법

- 소프트웨어 보안 테스트(SAST : Software Application Security Test)
- 실제 웹 서버가 작동하는(동적 상태) 상황에서 테스트(DAST : Dynamic Application Security Test)
- 소스코드 라이브러리 분석(SCA : Software Composition Analysis)

# CVE(정보 보안 취약점 코드)

- 공개적으로 알려진 소프트웨어 보안 취약점을 알리는 코드번호
- 웹, 앱, 소스 코드 등 다양한 원인에서 발생하는 취약점들을 게시

# 소프트웨어 보안 취약점 점검기준

- 입력데이터 검증 및 표현
- 보안기능(인증, 권한 관리, 접근 통제 등)
- 시간 및 상태(시간차 상태변동으로 인한 취약점이 있는지)
- 에러처리
- 코드오류(간결하고 버그가 없어야 함)
- 캡슐화(인가되지 않은 사용자의 접근 차단)
- API 도용(도용을 막을 수 있어야 함)

# 소프트웨어 개발보안 개념

- 안전한 SW개발을 위해 개발 생명주기(소프트웨어 개발 라이프사이클)에 맞추어 보완활동 추가

- 요구사항 분석 : 보안 관련 항목 식별 및 명세서 작성
- 설계 : 위협 모델링, 보안설계서 작성
- 구현 : 표준 코딩 정의서와 보안 가이드 준수, 보안약점 진단 및 개선
- 테스트 : 모의침투 또는 동전분석 실시
- 유지보수 : 지속적인 패치(보안패치)

# 에러 기반 공격기법

- 사이트에서 반환하는 에러 반응을 바탕으로 쿼리 문자 등을 조합하여 사이트 공격
- SAST에서 취약점 테스트 및 점검

# 널바이트 인젝션

- Java, C++의 경우 문자열의 끝에 '문자열 끝'을 의미하는 문자(%00, 0x00)가 들어간다.
- 파일 등을 전송할 때 파라미터의 끝을 '문자열 끝'으로 설정하여 확장자명을 숨길 수 있다.

# CSRF 토큰 방식의 보안 취약점

- Stored
- XSS Dom
- XSS Reflected(반사)

- 사이트마다 보안 취약점이 다르며, 취약한 정도의 경중에 따라 처리 방식이 달라질 수 있다.

# CSRF 크로스사이트 요청위조

- 게시글을 여러 개 작성한 후 파로스 툴을 이용해 웹 서버 작동 방식 확인
- 이후 CSRF 스크립트가 포함된 게시물을 등록하면 사용자 위/변조 및 사용자 명의를 도용한 행위 가능

# CRLF injection(HTTP 응답분할)

## 공격 사례

1. CSP태그 밀기

- CSP에서는 헤더에 입력하는 값 중 하나라도 유효하지 않으면 정상적인 응답을 하지 않는다(보안 취약점 노출 방지)
- 다만 헤더에 입력하는 값이 제대로 필터링된 값이 아닌 경우 응답이 분리되어 전달되므로 공격자가 요청을 변조할 수 있다.
- Java의 경우 로그(log)를 조작하는 사례도 있다.

- 이를 방지하려면 입력값의 null 여부를 체크하고, 응답헤더는 개행문자를 제거한 값을 사용해야 한다.

# 신뢰하지 않는 URL로 자동 접속

- 보안결정에 사용되는 값은 사용자 입력이 아닌 서버에 저장된 값으로 처리해야 한다.
- 회원정보 수정 시 요청한 사용자와 로그인한 사용자가 일치하는지 확인

# 파라미터 삽입

- 일반적으로 비공개된 개인 IP 주소는 공개적으로 접속할 수 없다.
- 하지만 공개된 URL에 파라미터 값으로 'url=비공개 IP 주소'를 삽입하여 보안 매커니즘을 우회할 수 있다.

- 화이트리스트 방식(가능한 것 빼고 모두 차단)을 사용하여 파라미터에 IP 주소 형식의 값을 입력하지 못하게 막아야 한다.

# 인증수행 제한

- 특정 계정에 대해 일정 횟수 이상 인증에 실패할 경우 실패 이력을 추적하고, 반복 인증 시도를 막아야 한다.
- Brute-Force(비밀번호 무차별 대입) 공격을 막기 위함

# 하드코드된 비밀번호

- 비밀번호 정보를 하드코딩으로(입력값 그대로) 저장 및 취급하면 인증 시스템에 문제가 생겼을 때 문제 파악 및 해결이 어려워지고, 보안 위협에 크게 노출되는 문제가 있다.

# 개인키와 공개키

- 대칭키 : 암호화와 복호화에 사용하는 키가 동일한 키(자물쇠와 열쇠의 한 쌍)
- 비대칭키 : 암호화와 복호화에 사용하는 키가 다른 경우, 개인 키와 공유 키로 나누어 관리한다.
  - 암호화 시 개인키를 사용하고 복호화 시 공유키를 사용하게 할 수 있고, 그 반대도 가능하다.
  
## 레인보우 테이블 어택

- 흔히 알려진 키 알고리즘(레인보우 테이블)을 하나씩 대입하여 어떤 암호화 방식인지 알아내는 방법
- 비밀번호 길이를 길게 하고, 예측하기 어렵게 설정하는 것이 좋다.

# 사설 인증

- 웹 사이트 인증서가 공식적이지 않은 경우 웹 브라우저에서 경고를 표시한다.
- 해당 웹 사이트에 접속할 경우 악성코드 배포, 해킹 등의 피해를 볼 수 있다.

# 에러 상황 기반 취약점

## 로깅(LOGGING) 처리

- 시스템 에러로그를 그대로 화면에 표시하지 말 것(보안 취약점 노출 가능성)
- 별도의 로그 기준을 마련한 뒤 해당 기준에 맞추어서 에러문을 표시하기

## 에러페이지 설정 부재

- 프로그램에서 어떤 일이 일어나고 있는지 전혀 알 수가 없다.
- 구체적이고 적절한 에러 메시지 반환은 보안을 위해 필요하다.

## 부적절한 예외처리

- 다양한 예외 기준이 발생했음에도 이를 세분화하지 않고 광범위한 클래스를 사용하는 경우
- 예상 가능한 예외들은 분기하여 표시하는 것이 좋다.
